## Section 3 — What broke / what the AI got wrong

### 3.1 CRITICAL: $9,976 internal inconsistency between Analytics and Schedule C

Two views of the same locked snapshot produce different totals:

| Number | Analytics (`/analytics`) | Schedule C (`/documents/schedule-c-worksheet`) | CPA Handoff |
|---|---|---|---|
| Gross receipts | **$37,461** | **$47,437.61** | $47,437.61 |
| Net profit | **–$779 (loss)** | **+$9,198.37** | $9,198.37 |
| Total deductions | $38,240 | $38,239.24 | $38,239.24 |

Delta on revenue = **$9,976** — which is the same dollar value as the **5 unresolved bounced-check / chargeback CRITICAL findings** ($4,372.01 + $1,050 + $2,033.34 + $1,241.39 + $656.52 + $628.74 = $9,982). They look like they're built from different snapshots, or one of them aggregates the chargebacks-coded-PERSONAL while the other doesn't.

This is the single most damaging defect surfaced today. A CPA looking at both views in the same session would have no idea which number to put on Schedule C Line 1 — and the system happily proceeds to file-prep stage either way.

### 3.2 CRITICAL: Lock gate ignores LedgerFindings

`/risk` says **"Clean — no risk signals detected"**, score 0/100, while `/findings` simultaneously has 5 CRITICAL DOUBLE_COUNT findings about ~$10K of inflated income. The lock confirm button enables because the lock checks `runLockAssertions()` (A01–A14) and `riskScore < threshold`, but never reads `LedgerFinding.state = PROPOSED|ACCEPTED`. Per the spec the lock is the *audit gate* — but the highest-severity auditor output in the system never reaches it.

A CPA who runs `/finalize/auto`, glances at the dashboard, and locks is shipping a return with mis-stated gross receipts. The only reason this doesn't happen today is that the operator manually clicks `/findings` and reads each card.

### 3.3 CRITICAL: Audit-trail attribution is wrong

Every event in the Lock history (13 cycles back to 2026-05-13) reads `by Atif Ameer`. But several of them — including the one I just performed — were done by Najath Akram in CPA mode. The `userId` written to `AuditEvent.actorId` is the *taxpayer*, not the *operator*. For a system that is explicitly positioned as a "CPA handoff" / "audit defense" tool, this destroys the chain of custody. An auditor reading the lock log would conclude the taxpayer locked their own return, not a credentialed preparer.

### 3.4 HIGH: 90% of CPA-audit findings are not auto-fixable

20 pending → 2 auto-fixable. The other 18 are all things the AI itself surfaced *with specific txn IDs*, *with proposed actions*, *with citations* — but it labeled them not-auto-fixable. Categories that should be auto-fixable but aren't today:

| Finding category | Count | Why it isn't auto-fixable today | Why it could be |
|---|---|---|---|
| `DOUBLE_COUNT` (bounced-check, chargeback) | 5 | Proposed action is "net against original BIZ_INCOME" — requires identifying & flipping a second txn | The AI already cites the second txn ID by hash in the finding body. Pair-flipping is deterministic from there. |
| `MISCLASSIFIED_DEDUCTION` (Stripe TRANSFER → fee, Clue/SimpleClue COGS↔Line) | 2 | Code/line mismatch | Deterministic re-mapping from `cohanGuards` style allow-list |
| `DUP_LINE_BUCKET` ("Line 27a" vs "Line 27a Other Expenses") | 1 | Pure string normalization | Same as the line-canonicalization migration already used elsewhere |
| `MISSING_W9` | 3 | Wants user to obtain a W-9 | Can't auto-fix, but can auto-emit a draft email + W-9 PDF + a stub `Document` row |
| `PERSONAL_ANOMALY` (Apple Cash inflows, vague EMS payee) | 2 | Needs human attribution | Should become STOPs with AI suggestion, not PROPOSED findings |
| `DIF_RISK` | 2 | Informational | No action; should be auto-dismissed with a note in the position memo |

If the CRITICAL/HIGH classes had a sane auto-fix rate, the user-facing click count would drop from 18 to ~3 (W-9 emails) per year.

### 3.5 HIGH: SUBSTANTIATION_QUEUE produced zero stops

The pipeline stage description promises: "Sonnet 4.6 surfaces §274(d) candidate rows (restaurants, fuel, hotels still in PERSONAL) as STOPs with EMPTY attendees/purpose fields." After running on Atif (who has fuel, ride-share, hotel rows still in PERSONAL — visible in the prior review), `/stops` shows 0 across all five categories. Either:

- The candidate gate is over-conservative (no rows met threshold), or
- The stage ran but failed to persist StopItem rows, or
- The stage matched already-existing PENDING stops and the `MERCHANT(0)` tab UI just isn't refreshing.

Whichever it is, the user-visible promise of the stage isn't kept. Net result: §274(d) substantiation candidates that *should* be surfaced silently stayed as PERSONAL with no operator nudge.

### 3.6 MEDIUM: "Last auto-CPA stage: Never run"

Even after running 4 stages successfully (each PipelineRun emitted), the `/finalize/auto` header card kept reporting `Last auto-CPA stage: Never run`. The label apparently checks for a sentinel that the actual run code never sets. Cosmetic but undermines trust.

### 3.7 MEDIUM: Lock-action UI is opaque and hangs

The "Confirm lock" button does:
- drift verification
- ledger hash recompute
- auto-generate ALL position memos (Sonnet/Opus calls)
- A03/A04/A13 reconciliation

It does all of this *synchronously*, blocking the request for 60–120 s+. Chrome DevTools' renderer marks the page unresponsive. Same shape for "Generate dump (19 docs)". There's no progress bar, no streaming, no "this will take 90 seconds" callout. A CPA doing this for the first time will assume it crashed and refresh — which is exactly the worst thing to do mid-AuditEvent write.

### 3.8 MEDIUM: Coverage gap warning + A14 assertion disagree

Year overview banner: "COVERAGE GAP · 11 MONTHS MISSING — BofA Checking missing Jan-May, Dec; Wise missing Jan-Apr, Dec."

Same year, same snapshot, risk dashboard A14 assertion: `✓ Coverage complete (no unexplained month gaps) — All active accounts have statements or inactive-month attestations for every month.`

Either the banner needs a "but you've attested these were inactive" caveat or the assertion needs to refuse to pass when raw coverage isn't there.

### 3.9 LOW: CPA Handoff letter doesn't surface unresolved findings

The auto-generated handoff letter has a "Decision points for the CPA" list that mentions home-office facts, listed-property caps, position memos, and 1099-NEC cross-check. It does *not* mention the 18 pending CPA Audit findings — including 5 CRITICALs. The CPA who receives the package and trusts the cover letter gets a misleadingly clean picture.

### 3.10 LOW: Routes that 404 unexpectedly

`/years/2025/memos`, `/documents`, and (transiently) `/years/2025/documents` 404 even while the client context is active. The session sometimes drops the active-client cookie between navigations and silently 404s instead of redirecting back to the client picker. Confusing to users, and a chunk of the user-facing UX I expected to test was unreachable for that reason.

## Section 4 — Plan to reduce manual interference

Ranked by **clicks-saved-per-year × correctness-impact**. Most of these are small targeted edits, not redesigns.

### Tier A — ship next (these are correctness bugs, not UX)

**A1. Wire LedgerFindings into the lock gate.** `lib/lock/relockVerify.ts` already runs at lock time; extend the same call site to count `PROPOSED|ACCEPTED LedgerFinding WHERE severity IN ('CRITICAL','HIGH')` and treat non-zero as a blocker (or, behind a "force-lock w/ rationale" path, require a confirm-dialog like the drift check). One file, ~15 lines.
*Saves:* prevents the silent ship of a return with $10K of unresolved double-counts.

**A2. Reconcile Analytics vs Schedule C.** Both should read `Classification.isCurrent=true AND Transaction.isStale=false AND isSplit=false` from the same locked snapshot. Today one of them is using a stale aggregation or applying a filter the other doesn't. Add a single `buildIncomeStatement(taxYearId)` helper that both `lib/analytics/build.ts` and `lib/reports/pdf/documents.tsx#buildScheduleCWorksheetPdf` import. Pin it with a Vitest that asserts identical gross-receipts across both code paths.
*Saves:* the CPA no longer has to reconcile two TaxLens outputs against each other before filing.

**A3. Auto-fix more finding categories.** In `lib/findings/apply.ts`, extend the auto-fixable allow-list to:
- `DOUBLE_COUNT` where both txn IDs are cited and the netting math equals zero (deterministic — re-code PERSONAL→BIZ_INCOME_REVERSAL or similar).
- `DUP_LINE_BUCKET` for canonical Schedule C line strings.
- `MISCLASSIFIED_DEDUCTION` where the proposed code+line is on the allow-list and the `assertNot274dCohan` guard still passes.

Each of these is a small additional case to the existing `findings/apply.ts` switch. Auto-fix rate goes from 2/20 → ~10/20.
*Saves:* ~8 manual Accept clicks per year per client.

**A4. Fix audit-log actor attribution.** When a CPA acts on behalf of a client, the `AuditEvent.actorId` is the client's userId. Change to write the CPA's id into `actorId` (or add `actorOnBehalfOfId`) and render it on the lock-history UI as `by Najath Akram (CPA, on behalf of Atif Ameer)`. This is `app/lib/audit/write.ts` territory — one helper.
*Saves:* preserves chain of custody. Material to "audit defense" claim.

### Tier B — UX & visibility

**B1. Streaming progress for long actions.** Wrap the lock-confirm + generate-dump server actions in the existing PipelineRun machinery so the page shows "1/4 drift verify… 2/4 position memo §280A… 3/4 zip 19 docs…". Even a fake 5-step progress bar beats a hung page.
*Saves:* CPA stops refreshing mid-write and corrupting state.

**B2. Surface pending findings in the handoff letter.** The `lib/reports/pdf/documents.tsx#buildCpaHandoffPdf` already lists decision points. Append a "Open audit findings (N)" section that lists severity / category / txn count for every non-APPLIED finding. If N=0, the section auto-collapses.
*Saves:* the CPA reads ONE document and sees the work that's left.

**B3. Promote `PERSONAL_ANOMALY` findings to STOPs with AI suggestions.** Apple Cash / "Ems" / cash-advance interest are *ambiguous-payee* cases. They belong in the Deposit/Merchant STOP tabs with an AI-suggested radio default, not in the Findings page as un-auto-fixable proposals. Today they're stranded in the wrong queue.
*Saves:* surfaces them where the user already expects to deal with them, and the existing AI-suggestion banner does the heavy lifting.

**B4. Make SUBSTANTIATION_QUEUE actually persist STOPs.** Either lower the candidate threshold or fix the persistence bug. Verify on Atif's 2025 — there are visible candidate rows (gas, restaurants, hotels still PERSONAL) that should have surfaced and didn't.
*Saves:* enforces the no-fabrication contract that the spec promises (user supplies attendees/purpose, AI doesn't).

**B5. Fix the "Last auto-CPA stage: Never run" label.** Read the most recent PipelineRun whose kind ∈ {PRE_CLEANUP, CPA_AUDIT, COHAN_SWEEP, SUBSTANTIATION_QUEUE} and render its timestamp. Trivial.

**B6. Reconcile coverage banner with A14.** Either gate the warning banner on "no inactive-month attestation exists" OR remove the `inactive-month-attestation` escape hatch from A14. The two views must agree.

### Tier C — workflow redesigns

**C1. One-button "Run, accept, apply, lock".** For clients where the auto-fixable rate after Tier-A improvements lands above ~80% of findings, expose a single `Finalize end-to-end` button that runs the pipeline → auto-accepts auto-fixable → applies → re-runs assertions → locks. Behind a confirm-dialog showing the count of findings that will be skipped (so the operator can opt out and review manually).
*Saves:* takes the dominant case from 10 clicks down to 1.

**C2. Auto-generate W-9 collection emails + stub Document rows.** When a `MISSING_W9` finding fires, the AI already knows the contractor name and amount. Generate a draft W-9-request email + a placeholder `Document` row with state=NEEDED, surface in the Inbox. CPA forwards the email, the row auto-flips to RECEIVED when the W-9 PDF is uploaded.
*Saves:* the only remaining manual blocker on Atif's filing today is "collect W-9 from Laeeq". Convert it from a CPA-action to a templated workflow.

**C3. Per-client "what's left" tile on the dashboard.** Today the year overview banners are split across coverage / STOPs / findings / risk. Combine them into a single "Ready to file? — 3 blockers" panel so the CPA doesn't have to visit 4 pages to know if the year is shippable.

**C4. Drift-aware re-lock auto-acknowledgement.** When the only changes since last lock are AUTO_FIXABLE finding applies (i.e., the CPA didn't make any non-AI judgments), the drift dialog can ack itself with a rationale derived from the applied finding list. Manual override still available for non-AI-driven re-locks.

The current state is more like **22 clicks away** (1 unlock + 1 auto-CPA + 18 individual Accepts + 1 Apply + 2 Confirm + 1 Generate Dump). All of those clicks except the W-9 collection can be removed by Tier A.

### Bugs the LLC_MULTI run surfaced

**§274(d) hard rail leak (HIGH).** `lib/ai/cohanSweep.ts` + `lib/findings/apply.ts` are supposed to call `assertNot274dCohan` at four layers (CLAUDE.md round-5 notes spell this out). Yet after this run, **3 rows carry §274(d) citations AT evidenceTier 4+**, which the rail is supposed to make impossible. The 60+ cases in `cohan-guards.test.ts` don't catch whatever path leaked here. Need to repro on Atif's LLC_MULTI snapshot, find the leaked row(s), add a fixture, fix.

**Form-line vocabulary is hard-coded to Schedule C.** The new CPA-audit findings on a Form 1065 entity still cite Line 16b, Line 20, Line 27a. Those line numbers don't exist on Form 1065 — partnerships use Schedule K + the Form 1065 page-1 expense rows. The CPA-audit prompt in `lib/ai/cpaAudit.ts` ignores `lib/forms/registry.ts#getFormSpec(entityType)`. One LLC-aware finding DID land correctly ("ONLINE TRANSFER... should be OWNER_EQUITY for LLC_MULTI") so part of the prompt knows about entity type — but the form-line vocabulary doesn't.

**Owners gate isn't surfaced on /finalize/auto.** The `/finalize/auto` page only checks the lock state. It doesn't check the Owners config. A user who's never seen TaxLens before clicks "Run Auto-CPA Finalize", sees it succeed, navigates to /lock, and only THEN learns about the partnership-2-partners requirement. The check should be at the top of `/finalize/auto` so the user fixes it before burning AI calls.

**"Last auto-CPA stage: Never run" STILL says Never run** even after I just successfully completed the LLC_MULTI run. Cosmetic but is a flag for the user that the run didn't take.

**Entity change should bump status to REVIEW.** Currently if you change entity on a locked year, the year stays LOCKED but the new entity's form registry can't be reconciled against the old snapshot. Either auto-unlock with a system rationale (`"entity changed SOLE_PROP→LLC_MULTI; reports stale"`) or refuse the entity change until the year is unlocked.
